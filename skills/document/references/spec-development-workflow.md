# Spec开发工作流指南

> 基于《松立研发AI开发规范》的完整Spec开发流程，从需求到上线的端到端闭环。

## 📋 工作流总览

```
需求澄清 → PRD编写 → 需求拆解 → 功能设计 → 测试设计 → 编码实现 → 测试验证 → 上线发布
    ↓          ↓          ↓          ↓          ↓          ↓          ↓          ↓
 需求文档   PRD文档   任务拆解   设计文档   测试用例   代码实现   测试报告   上线报告
    ↓          ↓          ↓          ↓          ↓          ↓          ↓          ↓
 GitLab Wiki 上传与同步
```

## 🔄 阶段一：需求澄清与PRD编写

### 1.1 需求澄清会议
**参与人员**：产品经理、技术负责人、业务方
**输入**：用户需求、市场调研、竞品分析
**输出**：需求背景、问题描述、验收标准

**会议议程**：
1. 需求背景说明（10分钟）
2. 问题讨论与澄清（30分钟）
3. 验收标准确认（15分钟）
4. 下一步行动计划（5分钟）

### 1.2 PRD文档编写
**使用工具**：`/document pm 生成`
**文档结构**：
```markdown
1. 需求背景
2. 目标与范围
3. 功能需求
4. 非功能需求
5. 验收标准
```

**关键检查点**：
- [ ] 需求描述清晰无歧义
- [ ] 验收标准可量化
- [ ] 功能范围明确界定
- [ ] 非功能需求完整

### 1.3 PRD评审
**评审流程**：
1. 产品内部评审（1小时）
2. 技术可行性评审（1小时）
3. 业务方确认（30分钟）
4. 评审意见处理（2小时）

**评审要点**：
- 需求完整性
- 技术可行性
- 业务价值
- 风险识别

## 🔄 阶段二：需求拆解与功能设计

### 2.1 需求拆解会议
**参与人员**：技术负责人、开发工程师、测试工程师
**工具**：`/document dev 生成`
**输出**：任务拆解清单、工作量评估

**拆解步骤**：
1. 功能模块划分（30分钟）
2. 接口边界定义（30分钟）
3. 数据模型设计（30分钟）
4. 工作量评估（30分钟）

### 2.2 功能设计文档
**文档类型**：
- 架构设计文档
- 详细设计文档
- 接口设计文档
- 数据库设计文档

**设计原则**：
1. **高内聚低耦合**：模块职责单一
2. **可扩展性**：支持功能扩展
3. **可维护性**：代码清晰易理解
4. **性能优先**：考虑性能影响

### 2.3 设计评审
**评审重点**：
- 架构合理性
- 技术选型正确性
- 扩展性考虑
- 性能预估
- 安全设计

**评审输出**：
- 设计评审记录
- 改进建议
- 风险识别

## 🔄 阶段三：测试设计与用例编写

### 3.1 测试计划制定
**参与人员**：测试负责人、开发工程师
**工具**：`/document test 生成`
**输出**：测试计划、测试策略

**测试类型**：
1. **功能测试**：验证功能正确性
2. **性能测试**：验证系统性能
3. **安全测试**：验证系统安全性
4. **兼容性测试**：验证环境兼容性

### 3.2 测试用例编写
**用例模板**：
```markdown
用例ID：TC-001
用例名称：用户注册成功
前置条件：系统正常运行
测试步骤：
  1. 访问注册页面
  2. 输入有效信息
  3. 点击注册
预期结果：注册成功，跳转到首页
优先级：高
```

**用例覆盖要求**：
- 正向场景：100%覆盖
- 逆向场景：关键异常覆盖
- 边界场景：所有边界值覆盖
- 兼容场景：主流环境覆盖

### 3.3 测试数据准备
**数据要求**：
1. **真实性**：接近生产数据
2. **多样性**：覆盖各种场景
3. **可复用性**：可重复使用
4. **安全性**：不包含敏感信息

## 🔄 阶段四：编码实现与单元测试

### 4.1 开发环境准备
**环境要求**：
- 开发工具统一
- 代码规范统一
- 依赖管理统一
- 配置管理统一

**分支策略**：
```
main (保护分支)
├── develop (开发分支)
│   ├── feature/user-register
│   ├── feature/order-create
│   └── fix/bug-123
└── release/v1.0.0
```

### 4.2 编码规范
**代码质量要求**：
1. **可读性**：命名清晰，注释完整
2. **可维护性**：结构清晰，逻辑简单
3. **可测试性**：便于单元测试
4. **安全性**：无安全漏洞

**提交规范**：
```
feat: 新增用户注册功能
fix: 修复登录失败问题
docs: 更新API文档
test: 添加单元测试
refactor: 重构用户服务
```

### 4.3 单元测试
**覆盖率要求**：
- 语句覆盖率：> 80%
- 分支覆盖率：> 70%
- 函数覆盖率：> 90%
- 行覆盖率：> 75%

**测试原则**：
1. **独立性**：测试之间不相互依赖
2. **可重复性**：每次运行结果一致
3. **及时性**：代码提交前必须通过测试
4. **全面性**：覆盖关键业务逻辑

## 🔄 阶段五：集成测试与系统测试

### 5.1 集成测试
**测试范围**：
- 模块间接口集成
- 系统间数据流转
- 业务流程完整性

**测试策略**：
1. **自底向上**：从底层模块开始测试
2. **自顶向下**：从用户界面开始测试
3. **核心路径优先**：先测试核心业务路径

### 5.2 系统测试
**测试类型**：
1. **功能测试**：验证系统功能
2. **性能测试**：验证系统性能
3. **安全测试**：验证系统安全
4. **兼容性测试**：验证环境兼容

**测试环境**：
- 开发环境：功能测试
- 测试环境：集成测试
- 预发环境：系统测试
- 性能环境：性能测试

### 5.3 缺陷管理
**缺陷流程**：
```
发现缺陷 → 录入系统 → 分配处理 → 修复验证 → 关闭缺陷
    ↓          ↓          ↓          ↓          ↓
  测试      测试      开发      测试      测试
```

**缺陷等级**：
- **致命**：系统崩溃、数据丢失（24小时内修复）
- **严重**：主要功能失效（48小时内修复）
- **一般**：次要功能问题（一周内修复）
- **轻微**：UI问题、建议（按版本修复）

## 🔄 阶段六：验收测试与上线发布

### 6.1 用户验收测试（UAT）
**参与人员**：产品经理、业务方、测试工程师
**测试重点**：
- 业务流程完整性
- 用户体验满意度
- 业务需求符合度

**验收标准**：
- 所有严重以上缺陷已修复
- 核心业务流程测试通过
- 性能指标达到要求
- 用户反馈满意度>90%

### 6.2 上线前准备
**检查清单**：
- [ ] 代码已合并到发布分支
- [ ] 数据库脚本已准备
- [ ] 配置文件已更新
- [ ] 监控告警已配置
- [ ] 回滚方案已准备
- [ ] 上线checklist已确认

**上线审批**：
1. 技术负责人审批
2. 产品经理审批
3. 业务方审批
4. 运维负责人审批

### 6.3 上线发布
**发布策略**：
1. **灰度发布**：先发布部分用户
2. **蓝绿部署**：新旧版本并行
3. **金丝雀发布**：逐步扩大范围

**发布后监控**：
1. **系统监控**：CPU、内存、磁盘
2. **业务监控**：成功率、响应时间
3. **用户监控**：用户反馈、错误率

## 🔄 阶段七：文档整理与知识沉淀

### 7.1 文档整理
**文档类型**：
1. **技术文档**：架构设计、API文档
2. **用户文档**：用户手册、操作指南
3. **运维文档**：部署手册、故障处理
4. **项目文档**：项目总结、经验分享

**文档要求**：
- 完整性：内容完整无缺失
- 准确性：信息准确无误
- 可读性：语言简洁易懂
- 及时性：随系统更新及时更新

### 7.2 知识沉淀
**沉淀方式**：
1. **技术分享**：定期组织技术分享
2. **代码Review**：优秀代码模板入库
3. **案例库**：典型问题解决方案
4. **最佳实践**：开发规范、流程优化

**使用工具**：
- GitLab Wiki：文档管理
- Confluence：知识库管理
- 钉钉文档：团队协作

## 📊 工作流质量控制

### 质量门禁
| 阶段 | 入口条件 | 出口条件 | 质量负责人 |
|------|----------|----------|------------|
| 需求澄清 | 需求明确 | PRD通过评审 | 产品经理 |
| 设计评审 | PRD通过 | 设计文档通过 | 技术负责人 |
| 编码实现 | 设计通过 | 单元测试通过 | 开发工程师 |
| 集成测试 | 单元测试通过 | 集成测试通过 | 测试工程师 |
| 系统测试 | 集成测试通过 | 系统测试通过 | 测试负责人 |
| 上线发布 | UAT通过 | 上线成功 | 项目经理 |

### 度量指标
| 指标 | 计算公式 | 目标值 | 测量频率 |
|------|----------|--------|----------|
| 需求澄清时间 | PRD完成时间 - 需求接收时间 | < 3天 | 每次需求 |
| 设计质量 | 设计缺陷数 / 功能点数 | < 0.1 | 每次设计 |
| 代码质量 | 代码缺陷数 / 千行代码 | < 5 | 每次发布 |
| 测试覆盖率 | 覆盖代码行数 / 总代码行数 | > 80% | 每次构建 |
| 上线成功率 | 成功上线次数 / 总上线次数 | > 95% | 每月统计 |
